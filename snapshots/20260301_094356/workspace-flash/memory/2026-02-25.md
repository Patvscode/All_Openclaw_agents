# 2026-02-25 — MiniMax Tool Calling Fix

## Problem
MiniMax-M2.5 outputs tool calls as plain text (either `<minimax:tool_call>` XML or `[TOOL_CALL]` JSON) that Ollama doesn't parse into structured `tool_calls`. OpenClaw sees plain text → sends raw tool call syntax to Telegram.

## Root Cause
Ollama parses tool calls by looking for `<tool_call>` XML tags (same format used by Qwen3). Our custom Modelfile template told the model to use plain JSON, which Ollama doesn't intercept.

## Fix: Adopt Qwen3-style `<tool_call>` format in MiniMax template
Ollama natively parses `<tool_call>{"name":"...", "arguments":{...}}</tool_call>` tags.

### BEFORE (broken) — Modelfile template:
```
{{ if .System }}]~!b[]~b]system
{{ .System }}{{ if .Tools }}

# Available Tools (JSON function calling)
You have access to these tools. To call a tool, respond with a JSON function call in this exact format:
{"name": "tool_name", "arguments": {"param": "value"}}

Tools:
{{ .Tools }}

IMPORTANT: Output tool calls as JSON, never as XML. Do not use <minimax:tool_call> tags.{{ end }}[e~[
{{ end }}{{ range .Messages }}{{ if eq .Role "user" }}]~b]user
{{ .Content }}[e~[
{{ else if eq .Role "assistant" }}]~b]ai
{{ .Content }}[e~[
{{ else if eq .Role "tool" }}]~b]tool
<response>{{ .Content }}</response>[e~[
{{ end }}{{ end }}]~b]ai
```

### AFTER (fix) — Modelfile template:
```
{{ if or .System .Tools }}]~!b[]~b]system
{{ if .System }}{{ .System }}
{{ end }}{{ if .Tools }}
# Tools

You may call one or more functions to assist with the user query.

You are provided with function signatures within <tools></tools> XML tags:
<tools>
{{ range .Tools }}{"type": "function", "function": {{ .Function }}}
{{ end }}</tools>

For each function call, return a json object with function name and arguments within <tool_call></tool_call> XML tags:
<tool_call>
{"name": <function-name>, "arguments": <args-json-object>}
</tool_call>{{ end }}[e~[
{{ end }}{{ range $i, $_ := .Messages }}{{ $last := eq (len (slice $.Messages $i)) 1 }}{{ if eq .Role "user" }}]~b]user
{{ .Content }}[e~[
{{ else if eq .Role "assistant" }}]~b]ai
{{ if .Content }}{{ .Content }}
{{ else if .ToolCalls }}<tool_call>
{{ range .ToolCalls }}{"name": "{{ .Function.Name }}", "arguments": {{ .Function.Arguments }}}
{{ end }}</tool_call>
{{ end }}{{ if not $last }}[e~[
{{ end }}{{ else if eq .Role "tool" }}]~b]user
<tool_response>
{{ .Content }}
</tool_response>[e~[
{{ end }}{{ end }}]~b]ai
```

### Key changes:
1. Tools presented in `<tools>` XML tags (like Qwen3)
2. Model instructed to use `<tool_call>` tags (Ollama parses these!)
3. `.ToolCalls` used in assistant template for multi-turn tool conversations
4. Tool responses use `]~b]user` role with `<tool_response>` wrapper (like Qwen3)
5. Removed "IMPORTANT: never use XML" instruction (we WANT XML now)

## Key Lesson
**Ollama parses `<tool_call>` XML tags from model output into structured `tool_calls`.** Any custom-imported GGUF model can get tool calling by using the Qwen3-style `<tool_call>` format in its Modelfile template. The model just needs to be instructed to output that format — Ollama handles the parsing.

Without this, Ollama treats tool call output as plain text content, which OpenClaw forwards as-is to the user.
